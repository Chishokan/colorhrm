# 給与・シフトアプリ 連携メモ（フェーズD 判断材料）

外部GAS「給与・シフトシステム」の実機レビュー結果と、Xサーバー版（PHP/MySQL）との連携方針。

## 1. 給与アプリは何をするか
塾講師の **シフト申請→承認（確定）→ 月次の時給計算 → 振込用一覧** を行うGAS Webアプリ。
- 別スプレッドシート（`SS_ID=1Wcmvg4...`）を自前DBに使用（Color HRM とは別ファイル）。
- 画面：講師ログイン/シフト可能登録、管理者のシフト入力・確定・**給与一覧（振込）**・講師マスタ。
- **PDF/メールは無し**。エクスポートは給与一覧の**テキストコピー**のみ。

## 2. 給与計算ロジック（移植は容易）
- シフト文字列 `HH:MM~HH:MM` → 稼働分。**運営分 = 総稼働分 − 授業分**（授業分は管理者入力）。
- `授業給与 = round(授業分/60 × 授業時給)` ／ `運営給与 = round(運営分/60 × 運営時給)`。
- 運営時給は一律 **1031**。授業時給は **カラー×部門の最高レート**（WageRates）。
- 交通費：勤務 ≤5日は `日数×200`、超過は `ceil(日数/5)×1000`。

## 3. ★最重要：Color HRM との連携が「スプレッドシート直読み」
給与アプリの「ColorHRMと同期」は、**Color HRM のスプレッドDB（`ColorHRM_DB`）を直接 openById して**、
`Staff`（name/colorRank/departments/email/isActive/usePayroll）と `WageRates`（color×department→rate）を読み、
自前の `講師マスタ` の **授業時給・Gmail** を更新する。**突合キーは「氏名」のみ**（安定IDなし）。

→ **問題**：Color HRM を **MySQL に移行済み**の今、PHP版はこのスプレッドシートを更新しない。
つまり給与アプリの同期は **古いスプレッドの値**を読むことになり、**実質連携が切れている**。

## 4. 連携の選択肢

### 案A：給与・シフトも PHP/MySQL へ再構築（単一DB・推奨の本命）
- 新テーブル：`pay_rates`（時給表）／`shift_applications`（申請）／`shift_days`（確定シフト：staff_id×年月日, 授業分/運営分）。
- 給与計算（上式）をサーバ側に実装。振込一覧（CSV/コピー）。
- 利点：**単一DBで完結**、氏名重複・改名・同期タイミング問題が消える、Color HRM の staff_id で安定連携。
- 規模：大（シフト申請/承認＋給与計算＋一覧の一機能群）。ただしロジックは単純で移植容易。

### 案B：給与アプリは外部GASのまま、PHPから「橋渡し」
- PHPが `Staff`＋`WageRates` を給与アプリへ供給（CSV出力 or スプレッド書き戻し）。
- 利点：給与アプリを作り直さない。欠点：**二重DB＋氏名キー同期の脆さ**が残る。Xサーバーからスプレッド書き戻しは Google API 認証が必要で重い。

### 案C：まず時給表だけPHPに載せ、給与アプリは当面リンクのみ
- `pay_rates` 取り込み＋ `staff.use_payroll` 表示＋外部アプリへのリンク（`payroll_url`）。
- 給与アプリは当面スプレッド運用のまま（同期は実質停止＝手動運用）。小さく始められる。

## 5. 推奨
Color HRM が MySQL 単一DBになった以上、**案A（PHPへ再構築）が一貫性の面で本命**。
ただし規模が大きいので、**段階導入**：
- **D-1**：`pay_rates`（WageRates 25件）取り込み＋時給表の表示/管理＋ staff の授業時給算出（カラー×部門）＋ `payroll_url` リンク。（小）
- **D-2**：シフト申請→承認→確定（shift_applications / shift_days）。（中）
- **D-3**：給与計算＋振込一覧（テキストコピー/CSV）。（中）
